# 📘 Day 11 — 데이터 모델링 / ER 다이어그램

## 0. 핵심 용어 빠른 참조

| 용어 | 설명 |
|------|------|
| 개체 (Entity) | 독립적인 의미를 가진 사람/사물/장소/개념 → 테이블 |
| 속성 (Attribute) | 개체가 가진 성질 → 컬럼 |
| 관계 (Relationship) | 개체 간의 연관성 → FK |
| 카디널리티 | 관계에 참여하는 개체 수 (1:1 / 1:N / N:M) |
| 강한 개체 | 독자적으로 존재 가능한 개체 |
| 약한 개체 | 상위 개체 없이 존재 불가 (이중 사각형으로 표시) |
| ISA 관계 | 상속 관계 — 학생 IS A 재학생/휴학생 |
| 전체 참여 | 모든 개체가 관계에 참여 (최솟값 ≥ 1) |
| 부분 참여 | 일부만 참여 가능 (최솟값 = 0) |

---

## 1. DB 생명주기

```
요구사항 수집·분석 → 설계 → 구현 → 운영 → 감시·개선
```

---

## 2. 3가지 모델링

| 단계 | 설명 | 산출물 |
|------|------|--------|
| 개념적 모델링 | 개체·관계 추출 | ER 다이어그램 |
| 논리적 모델링 | DBMS에 맞게 변환, 정규화 | 관계 스키마 |
| 물리적 모델링 | 실제 저장 구조 정의 | 테이블 생성, INDEX/VIEW |

---

## 3. ER 다이어그램 표기법

| 요소 | 도형 | 설명 |
|------|------|------|
| 개체 타입 | ☐ 사각형 | 테이블 |
| 속성 | ○ 타원 | 컬럼 |
| 키 속성 | ○ 타원 + 밑줄 | PK |
| 관계 | ◇ 마름모 | 테이블 간 관계 |
| 약한 개체 | ☐☐ 이중 사각형 | 상위 개체 필요 |
| ISA 관계 | ▽ 역삼각형 | 상속 관계 |

---

## 4. 카디널리티 (관계 대응수)

| 관계 | 설명 | 예시 |
|------|------|------|
| 1:1 | 각 개체가 1개씩 대응 | 학생 ↔ 학번 |
| 1:N | 한쪽이 여러 개 대응 | 학과 → 학생 |
| N:M | 양쪽 모두 여러 개 대응 | 학생 ↔ 강좌 |

> N:M 관계는 실제 구현 시 **중간 테이블(매핑 테이블)** 로 분리

---

## 5. 요구사항 분석 → ER 모델 변환 예시

### 도서관 시스템
```
요구사항: 회원이 도서를 대출/구매한다
→ 개체: 회원 (회원번호, 이름, 전화번호)
         도서 (ISBN, 제목, 저자)
→ 관계: 회원 -<대출>- 도서 (N:M)
         회원 -<구매>- 도서 (N:M)
```

### 쇼핑몰 주문 시스템
```
요구사항: 고객이 여러 상품을 주문, 주문 시 수량/날짜 기록
→ 개체: 고객 (ID, 이름, 이메일)
         상품 (상품ID, 상품명, 가격)
→ 관계: 고객 -<주문>- 상품 (N:M)
→ 매핑: 주문 테이블 (고객ID, 상품ID, 수량, 주문일)
```

### 인사관리 시스템
```
요구사항: 직원은 하나의 부서에 소속, 근태에 따라 급여 계산
→ 개체: 직원 / 부서 / 근태 / 급여
→ 관계: 직원 -<소속>- 부서 (N:1)
         직원 -<기록>- 근태 (1:N)
```

## 6. 전체 개발 흐름 정리

```
요구사항 분석
    ↓
ER 모델링 (개체·속성·관계 추출)
    ↓
논리적 설계 (정규화, 데이터형 지정)
    ↓
물리적 설계 (CREATE TABLE, INDEX, VIEW)
    ↓
JDBC 연동 (Java ↔ Oracle)
    ↓
웹 개발 (JSP / Spring-Boot / Node.js)
    ↓
배포 (AWS / Docker / CI-CD)
```
